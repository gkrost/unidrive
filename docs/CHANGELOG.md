### Fixed
- **`PlaceholderManager.dehydrate` produced fully-allocated NUL-byte files on JDK 21 jbrsdk Linux.** `RandomAccessFile.setLength(0)` then `setLength(N)` emits `ftruncate(0) + write(zeros, N)` on JDK 21 jbrsdk\_jcef-21.0.10 — a real page-allocating write — rather than the `ftruncate(N)` hole-punch JDK 25 emits. Strace evidence captured 2026-05-01. Same UD-712 incident class on a different OS. Switched to `FileChannel.open(WRITE, TRUNCATE_EXISTING) + position(N-1) + write(byteArrayOf(0))` so a single trailing 4 KiB page is allocated regardless of JDK. SPECS.md §0 placeholder-rationale paragraph updated to drop the now-falsified "Linux setLength is sparse" claim. [UD-209a, UD-758]
- **`LocalScanner` clobbered the UD-222 sparse-leftover invariant after UD-901.** UD-901's pending-upload row in LocalScanner unconditionally wrote `isHydrated=true` for every new on-disk file — including sparse leftovers from interrupted syncs. The engine then adopted those files as fully synced, never re-downloading; the user silently saw zero-byte content forever. UD-901's contract is preserved for the dense-file case; sparse leftovers (>4 KiB physical, blocks×512 < logical size) now correctly get `isHydrated=false`. Reconciler's `remoteState` computation also gained a special case so a UD-901 pending-upload row (`remoteId=null && remoteHash=null`) is treated as `ChangeState.NEW` rather than mis-classified as `MODIFIED` when remote-side delta later surfaces the path — the previous (NEW, MODIFIED) state pair had no resolveAction case and silently dropped the sync action. [UD-209b]
- **`IpcServerTest` four data-flow tests races between `runTest` virtual time and real Unix-domain-socket I/O.** `runTest` skipped `delay()` in virtual time while the server's accept loop, broadcast pump, and client `readFromClient` loop ran in real wall-clock — `emit()` raced the read deadline. Switched the four affected tests (`broadcast sends message…`, `stale socket file is reclaimed`, `late joiner receives state dump`, `state dump events include profile and timestamp fields`) to `runBlocking(Dispatchers.IO)` with an explicit `serverScope = CoroutineScope(coroutineContext + SupervisorJob())` so the accept loop can be cancelled at test end. The remaining 5 IpcServerTest cases assert on conditions that aren't timing-sensitive and stay on `runTest`. [UD-816]
- **`CloudRelocatorTest.UD-286 - per-file CancellationException propagates`** raced because the default `maxConcurrentTransfers = 4` let `cancelled.txt` and `never-reached.txt` launch in parallel, so `never-reached.txt` could complete upload before `cancelled.txt`'s CE fired `outerScope.cancel(e)`. Test now passes `maxConcurrentTransfers = 1` for deterministic CE-vs-sibling ordering. Production CE-propagation contract unchanged. [UD-817]

- **OneDrive `downloadFile` allocates `byte[contentLength]` for files > 2 GiB.** Symptom in a live `unidrive get /` test-drive: every file > ~2.147 GiB failed with `Can't create an array of size N` after 3 retry attempts; 5 of 40 files in a 23 GiB Videos/ tree were skipped. Root cause: `httpClient.get(url)` followed by `response.body<ByteReadChannel>()` is a buffered path on Ktor 3.x — the channel is exposed only after the entire body has been allocated as a single `byte[]`, and Java arrays are int-indexed (`Integer.MAX_VALUE` ≈ 2.147 GiB). Fixed by switching to `httpClient.prepareGet(url).execute { response -> ... }` and reading `response.bodyAsChannel()` inside the `execute { }` block — Ktor's blessed streaming path. The 401-refresh-once and 429/503 throttle handlers that lived in `authenticatedRequest` are now inlined into `downloadFile`'s retry loop via a new private `DownloadOutcome` sealed class (`Done | RetryAuth | Throttle`), because `execute { }` is the only way to scope a streaming body's lifetime correctly. Existing OneDrive tests stay green; live retest of the original 5-file failure profile is the acceptance criterion. [UD-329]

### Fixed
- **Sync resilience under real OneDrive traffic (UD-222 marathon, session 2026-04-19)** — seven interlocking fixes that together make a 346 GB / 130 k-item first-sync survivable end-to-end. (i) `PlaceholderManager` no longer pre-allocates NUL bytes; non-folder remote changes route through `DownloadContent` in Pass 2 [UD-222]. (ii) `Reconciler` synthesises `DownloadContent` for orphaned non-hydrated DB entries, rescuing files left behind by previous partial runs — without this the UD-222 cursor-promotion guard would falsely report success [UD-225]. (iii) `GraphApiService.authenticatedRequest` retries on 429/503 honouring `Retry-After` header [UD-207]. (iv) `downloadFile` CDN path also retries 429/503 and falls back to parsing `retryAfterSeconds` from the JSON error body when the header is absent [UD-227]. (v) `getDelta` retries on `SerializationException`/`IOException` from truncated Graph responses [UD-308]. (vi) `downloadFile` retries on Ktor "Content-Length mismatch" short reads [UD-309]. (vii) `authenticatedRequest` retries once on 401 after forcing a token refresh via a new `forceRefresh` hint on `tokenProvider`; refresh itself wrapped in `NonCancellable` so parent scope cancellation can't kill it mid-flight [UD-310]. Budget caps throughout (5 attempts, 2 min per retry, 10 min per request total). Live-tested: the same sync run that previously died on 2,060 × unhandled 429 now absorbs 487+ throttle waits with a single fatal survivor (picked up by UD-225 on the next run).
- `sync --download-only` no longer writes 346 GB of NUL-byte stubs on a 346 GB OneDrive. `PlaceholderManager.createPlaceholder` drops the `RandomAccessFile.setLength(size)` pre-allocation branch (NTFS fully allocates; Linux sparse was still visibly broken on file open). Reconciler routes remote-new/remote-modified non-folders through `DownloadContent` (Pass 2, concurrent) instead of `CreatePlaceholder(shouldHydrate=false)`, so the default is a real download. Pin rules no longer gate hydration (reserved for CfApi via UD-401/402/403). `pending_cursor` promotion now guards on transfer-failure count so partial first-sync failures retry on the next run; AuthenticationException is captured per-job and rethrown after scope exit instead of surfacing as `JobCancellationException`. 5 tests rewritten + new cursor-promotion-guard / Pass 2 failure-aggregation tests [UD-222].
- `RcloneProviderFactory` now normalises `rclone_remote` to guarantee a trailing colon before building `RcloneConfig`. Without this, TOML users writing `rclone_remote = "minio"` (matching the `rclone.conf` section-header convention) produced commands like `rclone lsjson miniopath` — every call exited with "directory not found". Discovered via the UD-803 live harness; regression locked in by 2 new factory tests [UD-306].
- `core/docker/docker-compose.integration.yml` — corrected bytemark/webdav env vars (`AUTHUSER`/`AUTHPASS` → `USERNAME`/`PASSWORD` + `AUTH_TYPE=Basic`) and removed the `/webdav/` URL suffix from the inline example. The file was unused by automation (only referenced for manual local testing) so the bug was invisible until UD-803 built a real harness against the same image.
