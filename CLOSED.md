## Drained from BACKLOG

- Internxt `encryptVersion` legacy support. Closed without implementing the assumed `02-rsa` decrypt path: an investigation across all current and historical public Internxt repositories (`inxt-js` back to v1.2.0, `sdk`, `drive-web`, `drive-desktop`, `drive-mobile`, `swift-core`, `cli`, `drive-server-wip`, `service-storage-models`, `core`, `node-lib`) found zero references to a `02-rsa` cipher format — `EncryptionVersion` enum in `@internxt/sdk` (`src/drive/storage/types.ts`) defines only `Aes03 = '03-aes'`, the server stores the column as a free string with no enum constraint, and every client unconditionally uses AES-256-CTR. The earlier README claim that "Vendor SDK supports `02-rsa` for legacy buckets" was inherited from an unverified assumption. Landed instead: a fail-fast guard in `InternxtProvider.downloadWithFileUuid` that throws `ProviderException("Unsupported encryptVersion '$version' for $remotePath…")` on any non-`03-aes` / non-null value, replacing the previous silent-garbage-bytes outcome with a clear error a user can file an issue about. Null encryptVersion is treated as `03-aes` (the field is absent on very-old records that predate the column). README updated to match the investigation result. If a legacy file with a real non-`03-aes` value ever surfaces, the error message tells the user to file an issue with the actual version string — at which point the format becomes discoverable and we can decide whether to implement.
- Internxt retry coverage on remaining mutating verbs (`putEncryptedShard` / `putEncryptedShardFromFile` / `downloadFileStreaming`). Three wraps with three different boundaries, each chosen by what the retry must hold steady. `putEncryptedShard` (in-memory, currently unused but retained for API surface stability) wraps `retryOnTransient` at the API service — idempotent at the OVH S3 layer, no caller-side state to pin. `putEncryptedShardFromFile` stays single-shot at the API service; the retry lives at the provider in a new private `retryShardCommit` helper that wraps stages 5-7 of `upload()` (startUpload → PUT → finishUpload) so the temp file, indexBytes, iv, fileKey, and hashHex all stay byte-identical across attempts. The wrap retries IOException and 5xx; never 4xx (logically-final). Capped at 3 attempts with simple exponential backoff (Retry-After honoured when present). startUpload is re-issued per attempt because OVH presigned URLs expire and a fresh descriptor is cheaper than wrapping the stale URL in an extra `getBridgeFileInfo` round-trip; shardUuid drift across attempts is safe because `commitWithRetry`'s 409 reconcile keys on plainName + size + start window, not the shardUuid. `downloadFileStreaming` also stays single-shot at the API service; the retry sits in the provider's `downloadWithFileUuid` and re-resolves the OVH download URL via `getBridgeFileInfo` plus re-constructs the `Cipher` per attempt (the existing one is single-use via `doFinal()`). Re-derived iv and fileKey are deterministic given the server-stored indexHex so re-derivation is safe. `TRUNCATE_EXISTING` on the destination open mode (unchanged) discards any partial prior attempt's bytes. **Also discharges the BACKLOG design constraint** "Internxt upload retry must pin indexBytes": option (a) chosen — retry stages 5-7 only, indexBytes computed once outside the retry boundary. One pinned IV-pinning test (`upload retry pins indexBytes across attempts (IV stability)`) added to `InternxtApiServiceTest`: drives a real `InternxtProvider.upload()` against a stateful MockEngine that fails attempt-1 PUT and attempt-2 finishUpload, then succeeds attempt-3. Asserts that across the two finishUpload calls the `indexHex` is byte-identical AND that the PUT ciphertext bodies are byte-identical across all three attempts (equal ciphertext under AES-CTR proves indexBytes/iv/fileKey all stable; any drift would change the keystream).

- Internxt parallel listing pagination. `InternxtProvider.delta()` now fans out 2 page fetches per stream and runs `/files` and `/folders` pagination concurrently rather than serially. Heartbeat correctness preserved via two `AtomicInteger` counters per stream that the helper bumps as pages arrive; the combined-total lambda passed through to the heartbeat reports a monotonic file+folder count without cross-coroutine list synchronization on the merge path. The 500/503 fallback to `collectFilesFromFolders` is unchanged in semantics — wrapped around the files stream's await so a `/files` outage still flips to the folder walk while the `/folders` stream continues (folders has no equivalent fallback so a `/folders` failure propagates and cancels the files stream via structured concurrency, matching the prior behaviour). No new tests: existing `delta()` coverage exercises the merged path and the `complete=false` accounting; this is a perf refactor, not a correctness change.
- Internxt 401 → automatic refresh-and-replay. Mirrors OneDrive's `GraphApiService.authenticatedRequest` ladder on the Drive REST surface so a mid-session JWT rotation no longer surfaces as a hard `AuthenticationException` to the user. `AuthService.getValidCredentials(forceRefresh: Boolean = false)` gains the same reactive knob OneDrive's `TokenManager.getValidToken` has — `forceRefresh = true` bypasses the `isJwtNearExpiry` short-circuit and goes through `refreshToken`, where `RefreshableTokenLatch` already coalesces concurrent forced refreshes into one HTTP round-trip. `InternxtApiService`'s constructor now takes `credentialsProvider: suspend (forceRefresh: Boolean) -> InternxtCredentials` and a private `withAuthRetry { creds -> ... }` helper sits between the `retryOnTransient` outer ladder and the per-attempt `driveBudget.awaitSlot()` on every Drive REST method (`createFile`, `replaceFile`, `createFoldersBatch`, `createFolder`, `renameFile`, `renameFolder`, `moveFile`, `moveFolder`, `deleteFile`, `trashItems`, `deleteFolder`, plus inside `authenticatedGet`'s loop body). Composition order is `retryOnTransient { withAuthRetry { budget.awaitSlot(); ... } }` — budget stays innermost so each retry attempt and each 401-replay re-acquire their own slot rather than holding the original attempt's slot through the auth refresh. `authenticatedGet`'s special shape composes the helper around the **interior** of one transient-retry iteration so a mid-call 401 reuses the current attempt's delay budget instead of restarting the 3-iteration ladder. - state.db redesign: tombstones, uuid identity, parent_uuid — drop-and-rescan upgrade. Three schema changes addressing the gaps surfaced by the cloud-trash recovery incident and the `.safe → .tresor` rename-descendants bug. (1) `status TEXT NOT NULL DEFAULT 'EXISTS'` with values EXISTS/TRASHED/DELETED; cloud deletes (engine-driven `applyDeleteRemote` and the reconciler's remote-reports-deleted cascade through `applyDeleteLocal`) flip via idempotent `UPDATE … WHERE status='EXISTS'` so the row survives as a queryable tombstone. The `RemoveEntry` apply path inspects the alive row at apply time: pending-upload row (synthetic `remote_id`) → hard delete; real `remote_id` → status flip. Move-source cleanup keeps `deleteEntry` because the item moved rather than disappeared. (2) PK swap from `path` to `remote_id` (the Internxt UUID; pending-upload rows get a `local:<uuid4>` synthetic at the storage layer that the row mapper translates back to Kotlin's nullable `remoteId`), with `path` as a partial unique index `WHERE status='EXISTS'` so trash-then-recreate at the same path is INSERT-clean. (3) `parent_uuid` column with composite `(parent_uuid, status)` index so "alive children of X" is an index seek; populated from `CloudItem.parentId` which Internxt fills from `folderUuid`/`parentUuid`. The sync loop reads only through the `alive_entries` view via `StateDatabase` methods; tombstones are reachable only via the `StateDatabase.recovery` namespace (compiler-enforced split — there is no method on `StateDatabase` itself that returns a non-EXISTS row). Upgrade is drop-and-rescan: a `schema_version` row in `sync_state` distinguishes pre-redesign DBs (no such row) from fresh and current installs; pre-redesign DBs drop `sync_entries` and the next scan repopulates. The prior one-shot `dedupe_remote_id` migration is subsumed and its tests are removed.
- Internxt resumable scan with persistent checkpoint. `InternxtProvider.delta()` now stages each retrieved page into a dedicated `scan_staging` table in `state.db` as it arrives and advances a `(filesOffset|foldersOffset)` checkpoint in `sync_state` after every successful page, so a daemon kill mid-scan (or the gateway's silent ~20k-item throttle) no longer wastes prior round-trips — the next launch resumes from the persisted boundary instead of restarting at offset 0. New `StateDatabase` primitives (`beginScan` / `getActiveScan` / `persistScanPage` / `loadStagedItems` / `completeScan`) own the slice + checkpoint lifecycle; `SyncEngine.gatherRemoteChanges` provisions a `ScanContext` (extended `CloudProvider.delta` parameter, defaulted to null so OneDrive and test stubs stay source-compatible), threads it into the provider's `delta()`, and clears the slice on successful return. Stale checkpoints older than `SCAN_CHECKPOINT_STALE_HOURS` (6h, matching Internxt's `updatedAt` change-detection window) are discarded and the scan starts fresh — Internxt's filter has no resolution past that gap so trusting an old marker would silently miss items that mutated during it. Staging uses a dedicated `scan_staging` table (option b from the design brief) rather than overloading `sync_entries` with a `pending_scan_id` column: per-page paths are partially-resolvable at stage time (Internxt's folder graph isn't complete until every page lands), so storing raw cloud identity (uuid + parent_uuid + plainName + size + modified + hash) and re-resolving paths from the rebuilt graph on resume is simpler than reasoning about half-resolved sync_entries rows leaking through the alive view. The `scan_staging` table is fully orthogonal to `sync_entries` — no view changes, no partial-unique-index changes, no `schema_version` bump (created via `CREATE TABLE IF NOT EXISTS` so existing v2 DBs gain it on the next `initialize()`). `InternxtProvider.delta()` gains a `PageBoundaryPersister` (per-page-boundary serialization for the parallel files+folders streams sharing one SQLite connection) and a `parseResumeOffsets` / `buildMarker` round-trip; the parallel speculative pagination loop accepts a `startOffset` and an `onPage` callback so resume + persistence drop in without disturbing the cancellation / heartbeat semantics that landed with the parallel-listing refactor. Resume rebuilds the folder graph from `scanContext.resumedItems` + freshly-fetched pages before re-resolving paths, so the engine receives a fully-coherent `DeltaPage` regardless of where the prior crash happened. New unit tests inside the existing files: `StateDatabaseTest` pins the staging lifecycle (begin / persist / load / complete / stale-discard / atomicity / orthogonality to the alive view / additive in-place migration on a pre-existing v2 DB); `SyncEngineTest` pins the engine-side wiring (fresh scan provisions a ScanContext, resume surfaces a non-null marker + resumed items, stale checkpoint clears, successful gather clears the slice, mid-scan throw leaves both slice + checkpoint intact for the next launch); `InternxtProvider_DeltaPathResolutionTest` pins the helpers (marker round-trip, malformed marker degrades to (0,0), root-parent normalization, file/folder re-inflation, resumed folder feeds back into `buildFolderPath`). Streaming reconciliation (process pages as they arrive, don't wait for full scan) remains out of scope per the BACKLOG note and gets its own spec later — this work is purely the per-page durability + resume win.
- Internxt cross-session resumable scan. `SyncEngine.gatherRemoteChanges` now skips `db.completeScan(scanId)` when the gather pass returned `complete=false`, so the `scan_in_progress_*` checkpoint (id + marker + started_at) and the `scan_staging` rows survive the clean exit and the next daemon launch's `getActiveScan` surfaces them as a resume hint instead of starting from offset=0. Pairs with the just-landed best-effort `delta_cursor` advance: the cursor moves forward by `max(updatedAt)` over completed pages while the offset doesn't regress, so a throttle-cliff'd account makes monotonic progress across daemon restarts — each restart picks up where the prior session got to, and eventually reaches `complete=true` even if no individual run does. The within-session checkpoint write cadence (per-page via `PageBoundaryPersister`) is unchanged; only the lifecycle of the completion clear changes. Safety net is the existing `SCAN_CHECKPOINT_STALE_HOURS = 6L` stale-threshold inside `getActiveScan` (already in `SyncEngine.kt` companion; not duplicated): a marker older than the threshold is treated as untrustworthy because Internxt's `updatedAt` change-detection window has lapsed, and the next launch starts fresh after clearing the stale state. `--reset` continues to wipe the marker via the existing `resetAll()` path (it deletes `sync_state`, which includes the `scan_in_progress_*` keys; orphan `scan_staging` rows are inert because the next `beginScan` mints a fresh scan_id). No SPI changes — `ScanContext` shape, `CloudProvider.delta` signature, and OneDrive's snapshot/`@odata.nextLink`-resume path are untouched (OneDrive ignores the marker as before). Two new unit tests inside `SyncEngineTest`: `gather preserves the checkpoint when delta returns complete=false` pins the cross-session preservation seam (incomplete clean-exit leaves marker + staged rows + started_at in place), and `cross-session resume clears the checkpoint once delta returns complete=true` pins the closing-of-the-loop (a pre-seeded cross-session checkpoint surfaces as `ScanContext.resumeMarker` + `resumedItems`, and a subsequent `complete=true` gather clears everything for the next-next launch). Existing `gather completes the scan checkpoint after a successful delta`, `gather leaves staging in place when delta throws mid-scan`, and the `StateDatabaseTest` staging + stale-discard suite remain green.
- Internxt best-effort cursor advance on incomplete scan. `SyncEngine.promotePendingCursor` (renamed from `promotePendingCursorIfComplete`) now writes `pending_cursor → delta_cursor` unconditionally — including when the gather pass returned `complete=false` because of the existing 500/503 subtree-skip fallback. The Internxt provider's `delta()` already computes the page cursor as `max(updatedAt) seen across successfully-fetched pages` (skipped folders contribute nothing because their items never enter `allFiles`/`allFolders`), so what's persisted is the freshest timestamp we can trust. The earlier behavior — pinning the cursor at its prior value on any `complete=false` — forced the next launch to re-query from the prior cursor (effectively a full re-scan on a hot 190k-item account) every time any folder hit a transient 503. Accepted correctness trade documented at the function's KDoc: items modified in the skipped subtree between the new cursor and the moment of the skip, that didn't change again afterward, will be missed until they next mutate — strictly no worse than the prior behavior, which missed the same items AND wasted hours re-fetching the rest. The `pending_cursor_complete` flag is still written so `DoctorCommand` continues to warn the user when the last sweep was incomplete and `--reset` is the existing escape hatch. Monotonicity floor lives in `InternxtProvider.advanceCursor` (companion helper, mirrors the `buildMarker` / `parseResumeOffsets` pattern): the returned cursor is `max(seenMax, requestCursor)` so an empty page-set or a stale-page race can't regress below the prior cursor. ISO-8601 lexicographic compare is sound because all Internxt `updatedAt` values share the `Z` zone suffix. OneDrive cursors are opaque URL strings — its monotonicity is provider-internal and untouched. Existing `incomplete delta gather` SyncEngine test renamed and flipped from pinned-buggy to pinned-correct (still pins the complete-flag write, the cursor advance under both complete=true and complete=false, and the recovery transition); five new unit tests inside `InternxtProvider_DeltaPathResolutionTest` pin `advanceCursor`'s four branches (advances on fresh, floors on empty/stale, falls back on first-scan-no-items).
- Internxt WebSocket change feed (wake-signal integration). A new `NotificationsClient` (`core/providers/internxt/src/main/kotlin/.../NotificationsClient.kt`) maintains a socket.io connection to the configured `NOTIFICATIONS_URL` with `auth: { token: jwt }` and listens on the `'event'` channel. Each remote-origin frame is treated as a coarse "something changed, walk the delta" wake hint — never as authoritative content, mirroring drive-desktop's `processWebSocketEvent` exactly. Loopback elision drops frames whose `clientId` matches `InternxtConfig.clientName` (default `"unidrive"`, the existing configurable header value sent on every Drive REST mutation — no new `'drive-desktop-unidrive'` literal introduced). Malformed / schema-incomplete frames fall through to "treat as remote" per the drive-desktop precedent so a missing field can't silently miss a change. `CloudProvider` gains one defaulted hook, `fun onRemoteChangeHint(callback: () -> Unit)` (no-op default; OneDrive inherits it untouched), and `SyncEngine` exposes `registerRemoteWakeListener(scope, listener)` which wraps the provider callback with a 5-second cancel-and-restart debounce job so a burst of 50 frames from a single folder-tree mutation coalesces into exactly one downstream wake. `SyncCommand`'s watch loop selects between the existing `LocalWatcher.awaitChange` and a new `CONFLATED` remote-wake channel so a debounced remote hint wakes the next sync pass early without otherwise touching the adaptive-poll math. `InternxtProvider.authenticate()` constructs the client lazily once a JWT is available; `close()` and `logout()` both call `NotificationsClient.disconnect()` to release the socket.io background thread. On `connect_error` (audit's named runtime gap, typically a JWT-rotation reject), the client coalesces concurrent refresh attempts, forces a fresh JWT via the existing `AuthService.getValidCredentials(forceRefresh = true)`, and rebuilds the socket; if refresh itself fails the client logs once at WARN and stays disconnected so the periodic poll (untouched safety-net per the audit) carries on. `NOTIFICATIONS_URL` is env-overridable via `INTERNXT_NOTIFICATIONS_URL` with the audit's best-guess default `https://notifications.internxt.com`, mirroring the existing `CRYPTO_KEY` / `INTERNXT_CLIENT_NAME` shape so an operator can substitute the production value (confirmed via mitmproxy) without rebuilding. New dependency: `io.socket:socket.io-client:2.1.2` (MIT) — transitive `engine.io-client:2.1.0`, `okhttp:3.12.12`, `okio:1.15.0 → 3.16.0` (already at 3.16.0 transitively from Ktor), `org.json:20090211`; no conflicts with the existing Ktor / OkHttp surface. The audit's "if real" branch noted post-MVP — promoted to MVP because the throttle is a present-day pain on user accounts.
- Internxt `finishUpload` idempotency via 409 reconcile. A network drop after the server-side commit but before the client reads the response previously caused a re-finish that orphan-billed a second `BucketEntry`; the Bridge exposes no client-addressable session-status endpoint to disambiguate. The fix layers a new internal `commitWithRetry(api, bucket, folderUuid, plainName, ext, fileSize, encryptedSize, indexHex, hashHex, shardUuid, startedAt, clock)` helper at the bottom of `InternxtProvider.kt` around the bridge `POST /v2/buckets/{bucket}/files/finish` call so the provider's upload pipeline now calls the helper at stages 7-8 instead of `api.finishUpload` directly. On a 409 whose body substring-matches either `"MissingUploadsError"` or `"Missing uploads"` (lenient OR locked by the first wire-shape observed: `{"statusCode":409,"message":"MissingUploadsError: …","error":"Conflict"}`), the helper lists the parent via `getFolderContents(folderUuid)` and filters its `files` for `plainName == plainName && type == ext && size.toLongOrNull() == fileSize && status == "EXISTS" && creationInstant >= startedAt - 5min && fileId != null`. Zero matches with no name match → case (a), re-attempt `finishUpload` once and surface a wrapped `InternxtApiException(statusCode = 409, cause = original409)` on the second 409. Zero matches but a stale name match exists → case (b), `delay(2s)` and re-list once before falling through to case (a). One match → `ReconcileResult.Found`, return the synthesized `BucketEntry(id = file.fileId, ...)`. Multiple matches → pick the youngest by `creationInstant` with a structured WARN naming both the chosen and rejected sibling fileIds (uses an SLF4J `ListAppender` in the new tests to assert the log shape). A listing failure during reconcile rewraps as `InternxtApiException(statusCode = 409, requestId = original.requestId, cause = listingErr)` so operators see both the original commit-side and the reconcile-side context. The reconcile window is hard-coded to 5 minutes from the upload's stage-1 `Instant.now()` capture so a slow retry that takes minutes to surface still matches its own committed file without colliding with stale entries hours-old. IV-pinning is honoured because the reconcile path holds `indexHex/hashHex/shardUuid` identity stable across the probe and the case-(a) re-attempt — same Bridge identity decrypts with the same key/IV as the original attempt. Case-(a) re-encrypt is intentionally not done at the helper layer (the helper has no local-file reach); the next natural sync cycle picks up a hard failure with fresh indexBytes. The bridge POST surface grows an optional `headers: Map<String,String> = emptyMap()` parameter (defaulted to empty for every existing caller) and `finishUpload` grows an optional `idempotencyKey: String?` parameter wired through to a one-line TODO comment so the eventual vendor-side `Idempotency-Key` header is a one-line change at the call site. Temp-file deletion moves from the stage-6 `finally` to a wider try/finally bracketing stages 6 through the post-commit return so the ciphertext survives the reconcile probe. Six new unit tests inside `InternxtApiServiceTest.kt` pin the helper's six branches via stateful URL-keyed MockEngines: happy-path single finish, 409→reconcile→single-match returns the listing's `fileId`, 409→reconcile→empty-listing re-attempts finish once then throws with cause, 409→reconcile→two size-matching candidates picks the youngest and WARN-logs the rejected sibling, 409→reconcile→listing 500 wraps the listing error as cause, and non-MissingUploads 409 propagates untouched without consulting the listing.
- Internxt local chunk-tombstone for upload resume. Compensates for the missing resumable-upload protocol so a daemon kill between encrypt and `finishUpload` no longer wastes the entire upload's prior work — the next `upload()` call for the same local path picks up at the recorded stage and reuses the pinned `indexBytes` / `iv` / `fileKey` / ciphertext / hash. A new internal `UploadTombstoneStore` (`core/providers/internxt/src/main/kotlin/.../UploadTombstoneStore.kt`) owns a sidecar JSON-per-upload at `${config.tokenPath}/upload-tombstones/<sha256(localPath)>.json` plus a co-located encrypted-ciphertext temp file at `<sha256(localPath)>.enc`. Sidecar shape (snake-case JSON, schema-discriminated): `local_path`, `local_mtime_millis`, `local_size`, `bucket`, `folder_uuid`, `plain_name`, `ext`, `index_bytes_hex` (32-byte server-stored salt that derives the IV via the mnemonic but is itself not a key — the derived `fileKey` / mnemonic / IV are NEVER persisted), `shard_uuid`, `bridge_put_url`, `encrypted_size`, `hash_hex`, `existing_remote_id`, `stage` (ENCRYPTING / PUT_PENDING / PUT_DONE / FINISH_DONE), `started_at_millis` (the canonical upload start, threaded into `commitWithRetry.startedAt` on resume so the 5-minute 409-reconcile window floors below the ORIGINAL attempt, not "now"), and `tombstone_written_at_millis` (refreshed on every transition; gates the URL-TTL freshness check). Atomic writes via `Files.move(tmp, final, ATOMIC_MOVE)` with a non-atomic `REPLACE_EXISTING` fallback for filesystems that reject the atomic flavour. POSIX 0600 best-effort where the platform supports it; no-op on Windows. `InternxtProvider.upload()` is restructured so it (a) GC-sweeps stale tombstones (missing-local or > `RESUME_TTL_MS = 7d` old) inline at the top of every upload, (b) computes a deterministic `pathHash` + .enc location and reads any existing tombstone, (c) validates the tombstone against the call shape (`local_path` + `local_mtime_millis` + `local_size` + `existing_remote_id` match, freshness within `RESUME_TTL_MS`) and discards-and-rotates `indexBytes` on any mismatch, (d) writes a stage=ENCRYPTING tombstone BEFORE touching the ciphertext so a crash mid-encrypt resumes with the same `indexBytes` and re-encrypts to the same deterministic .enc, (e) at the top of `retryShardCommit` consults the tombstone stage to skip startUpload / PUT / finishUpload on resume from each successive stage, (f) re-issues `startUpload` when the cached presigned PUT URL is older than `URL_TTL_MS = 10min` (the OVH ~15-min TTL minus a 5-min safety margin), (g) on success deletes both the .json sidecar and the .enc only AFTER the drive metadata write lands. Stage transitions inside the retry loop now also benefit from this — an in-process retry no longer redoes startUpload+PUT on every iteration; the existing `upload retry pins indexBytes across attempts (IV stability)` test was updated to pin the new shape (startUpload=1, PUT=2, finishUpload=2 instead of 3/3/2) while keeping its load-bearing IV-pinning assertions (indexHex byte-identity across the two finishUpload bodies; PUT-1 ciphertext == PUT-2 ciphertext under AES-CTR). Mid-upload `NoSuchFileException` (local file deleted between scan and upload) discards the tombstone before propagating. Two new TTL constants in `InternxtConfig.companion` (`RESUME_TTL_MS`, `URL_TTL_MS`). No `StateDatabase` touch, no SPI changes, no schema bump — the tombstone surface is filesystem-only. Eleven new unit tests inside `InternxtApiServiceTest.kt` pin the eleven branches: round-trip write/read; resume at each of ENCRYPTING / PUT_PENDING / PUT_DONE / FINISH_DONE; stale-on-mtime-drift; stale-on-size-drift; **stale-on-URL-TTL re-issues `startUpload` but preserves `indexBytes` (load-bearing IV-pinning regression — the entire reason this work exists)**; stale-on-existingRemoteId-mismatch; 409-reconcile on resumed `finishUpload` uses the original `startedAt` instant (proves the tombstone-anchored window, not "now"-anchored); missing local path during resume cleans up the tombstone and .enc.
