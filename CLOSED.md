## Drained from BACKLOG

- Internxt delta path-collapse (phantom folders). `buildFolderPath` now returns `String?`; null on missing non-root ancestor propagates through `toDeltaCloudItem` (also nullable) into `delta()`, which filters dropped items via `filterNotNull`, counts them, logs the count, and returns `DeltaPage(complete=false)` so the engine skips `detectMissingAfterFullSync` and leaves the cursor un-advanced. Regression test flipped from pinned-buggy to pinned-correct.
- Internxt state-db duplicate-`remote_id` cleanup. One-shot migration in `StateDatabase.createTables()` runs on first `initialize()` after upgrade. For each `remote_id` with more than one row, the row with the longest path wins (the bug shallowed paths, so the deepest is the pre-bug truth); the rest are deleted in a single SQL window-function DELETE. Idempotent: a `migration:dedupe_remote_id` marker in `sync_state` carries the deletion count and gates re-runs. Fresh databases set the marker with zero rows touched.
- Internxt listing `status` filter selectable per call. `InternxtApiService.listingQueryParams` / `listFiles` / `listFolders` now accept a `status: String` parameter defaulting to `"ALL"`; existing call sites preserve current semantics because they don't pass the new argument. Callers that need to scope a listing to `EXISTS`, `TRASHED`, or `DELETED` can now do so without rewriting the query map.
- Internxt `Retry-After` HTTP header parsing. `checkResponse` now reads the `Retry-After` response header via a new `parseRetryAfterHeader` helper that handles RFC 7231 §7.1.3's two forms — non-negative delta-seconds and IMF-fixdate (RFC 1123) HTTP-date — and stashes the parsed milliseconds on `InternxtApiException.retryAfterMs`. `retryOnTransient` prefers the header value over the JSON-body `retry_after` hint (header is the canonical signal per spec) and falls back to the body hint then exponential backoff. Header values pointing to a past date, malformed strings, or absent headers return null so the precedence chain falls through cleanly. The remaining "route DELETE mutations through `retryOnTransient`" sub-task stays open under its own BACKLOG entry.
- Internxt JWT pre-expiry refresh margin. New `AuthService.isJwtNearExpiry(thresholdMs)` mirrors OneDrive's `Token.isNearExpiry` shape and is wired into `getValidCredentials` with a 1-day margin (`JWT_REFRESH_MARGIN_MS`). A token within one day of its `exp` claim is now refreshed eagerly, so a long-running sync that starts with a five-minute-remaining token rotates *now* rather than tripping the un-replayed 401 path mid-stream and surfacing an interactive prompt. `isJwtExpired` stays as-is for the interactive re-auth gate at `InternxtProvider.authenticate`; the new helper is a strict superset (already-expired tokens still return true). The existing "still valid → no refresh" regression test was updated to seed a JWT 10 days out, outside the new margin, so it pins the same invariant under the new behaviour.
- Internxt `NonCancellable` wrap on `refreshToken`. Verified — already covered by `RefreshableTokenLatch.withRefresh`; the wrap encompasses `fetchRefreshedJwt` + `saveCredentials`. `AuthService.refreshToken` passes a `body` lambda to the latch that calls `fetchRefreshedJwt`, then `saveCredentials`, then assigns the in-memory `credentials` field — and the latch wraps the entire `body()` invocation in `withContext(NonCancellable)` (`RefreshableTokenLatch.kt:68`). A parent-scope cancellation between the network call and the disk write therefore cannot tear the refresh in half. The BACKLOG entry predated the UD-338 lift of this pattern into `:app:core`.
- Internxt in-flight dedup extended beyond `getFolderContents`. `InternxtApiService` now installs three additional `InFlightDedup` instances — `fileMetaDedup`, `listFilesDedup`, `listFoldersDedup` — mirroring the original `folderContentsDedup` wiring (one dedup per call site, per the `InFlightDedup` KDoc key-design guidance, no new abstraction layer). `getFileMeta` keys on `uuid`; `listFiles` and `listFolders` key on a `"updatedAt|limit|offset|status"` composite so concurrent callers walking the same cursor page collapse onto a single HTTP round-trip. Failure propagation, cancellation semantics, and post-completion eviction all come from the existing primitive untouched. The "request prioritization" sub-task remains open under a re-scoped BACKLOG entry.
- Internxt hard-coded client-header values now configurable. `InternxtConfig` grows three new fields — `clientName` / `clientVersion` / `desktopHeader` — that back the `internxt-client` / `internxt-version` / `x-internxt-desktop-header` triplet emitted by `applyInternxtHeaders`. Defaults preserve the historical wire shape; env-var overrides (`INTERNXT_CLIENT_NAME`, `INTERNXT_CLIENT_VERSION`, `INTERNXT_DESKTOP_HEADER`) plus direct constructor overrides let an operator rotate them without rebuilding when upstream tightens its allowlist. Pattern matches the existing `CRYPTO_KEY` override knob in the same companion. `applyInternxtHeaders` now takes the config as a parameter; the four call sites (one in `InternxtApiService.applyAuth`, three in `AuthService` for login-challenge / login-access / refresh) all already had a config reference in scope so the threading is purely mechanical.
- Internxt `HttpRetryBudget` wiring. `InternxtApiService` now owns two per-host `HttpRetryBudget` instances mirroring drive-desktop's audited `bottleneck` config: a Drive REST budget (`maxConcurrency=2`, `minSpacing=500ms`, storm spacing `1000ms`) and a Bridge upload budget (`maxConcurrency=4`, `minSpacing=0`, concurrency-only — the reference client adds no spacing here). Every HTTP-bearing method now acquires a slot from its appropriate budget per attempt (inside `retryOnTransient`'s lambda where present so the slot isn't held through backoff), records success on the 2xx path, and feeds 429/503 back via `recordThrottle` so the storm detector observes them. `authenticatedGet`'s manual retry loop also acquires per attempt and calls `recordIoRetry` on the EOFException path. The DELETE call sites that still lack `retryOnTransient` (tracked in `Internxt retry coverage on remaining mutating verbs`) acquire-and-record inline so the budget still sees their traffic. Both budgets default in the constructor so the existing test fixtures (and unit tests that don't care about coordination) keep their 2-arg call shape. A one-line code comment near each acquisition flags the future insertion point for the `withAuthRetry` wrapper that the 401 → automatic refresh-and-replay item will introduce — composition becomes `retryOnTransient { withAuthRetry { budget.awaitSlot(); ... } }` so the budget stays innermost and retries re-acquire a fresh per-attempt slot.
- Internxt parallel listing pagination. `InternxtProvider.delta()` now fans out 2 page fetches per stream and runs `/files` and `/folders` pagination concurrently rather than serially. A private `speculativeFetchPages` helper keeps a sliding window of two in-flight `fetchPage` calls per stream — matching the Drive `HttpRetryBudget` concurrency cap so the safety brake is honoured — and cancels speculative siblings when a short page arrives rather than waiting on a wasted round-trip (one cancelled fetch is much cheaper than one extra 28-52 s wall-clock window). Heartbeat correctness preserved via two `AtomicInteger` counters per stream that the helper bumps as pages arrive; the combined-total lambda passed through to the heartbeat reports a monotonic file+folder count without cross-coroutine list synchronization on the merge path. The 500/503 fallback to `collectFilesFromFolders` is unchanged in semantics — wrapped around the files stream's await so a `/files` outage still flips to the folder walk while the `/folders` stream continues (folders has no equivalent fallback so a `/folders` failure propagates and cancels the files stream via structured concurrency, matching the prior behaviour). No new tests: existing `delta()` coverage exercises the merged path and the `complete=false` accounting; this is a perf refactor, not a correctness change.
- Internxt 401 → automatic refresh-and-replay. Mirrors OneDrive's `GraphApiService.authenticatedRequest` ladder on the Drive REST surface so a mid-session JWT rotation no longer surfaces as a hard `AuthenticationException` to the user. `AuthService.getValidCredentials(forceRefresh: Boolean = false)` gains the same reactive knob OneDrive's `TokenManager.getValidToken` has — `forceRefresh = true` bypasses the `isJwtNearExpiry` short-circuit and goes through `refreshToken`, where `RefreshableTokenLatch` already coalesces concurrent forced refreshes into one HTTP round-trip. `InternxtApiService`'s constructor now takes `credentialsProvider: suspend (forceRefresh: Boolean) -> InternxtCredentials` and a private `withAuthRetry { creds -> ... }` helper sits between the `retryOnTransient` outer ladder and the per-attempt `driveBudget.awaitSlot()` on every Drive REST method (`createFile`, `replaceFile`, `createFoldersBatch`, `createFolder`, `renameFile`, `renameFolder`, `moveFile`, `moveFolder`, `deleteFile`, `trashItems`, `deleteFolder`, plus inside `authenticatedGet`'s loop body). Composition order is `retryOnTransient { withAuthRetry { budget.awaitSlot(); ... } }` — budget stays innermost so each retry attempt and each 401-replay re-acquire their own slot rather than holding the original attempt's slot through the auth refresh. `authenticatedGet`'s special shape composes the helper around the **interior** of one transient-retry iteration so a mid-call 401 reuses the current attempt's delay budget instead of restarting the 3-iteration ladder. Bridge calls (`bridgeGet` / `bridgePost` and the upload/download surfaces threaded under them) intentionally remain unwrapped — they use HTTP Basic auth derived from `bridgeUser` / `bridgeUserId`, not JWT, so a JWT refresh wouldn't help on a Bridge 401. A second consecutive 401 (one refresh, one still-rejected replay) propagates — both halves mean the auth flow itself is broken, not a transient. Two pinned assertions added inside the existing test files (no new test files, per the smoke-cap rule and the BACKLOG brief): `InternxtAuthServiceTest` pins that `getValidCredentials(forceRefresh = true)` triggers exactly one refresh roundtrip even with a fresh JWT, and `InternxtApiServiceTest` pins that a 401-then-200 MockEngine sequence on `createFolder` results in exactly two HTTP attempts plus exactly one forced-refresh callback (no second refresh on the 200).
