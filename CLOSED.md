- Internxt retry coverage on remaining mutating verbs (`putEncryptedShard` / `putEncryptedShardFromFile` / `downloadFileStreaming`). Three wraps with three different boundaries, each chosen by what the retry must hold steady. `putEncryptedShard` (in-memory, currently unused but retained for API surface stability) wraps `retryOnTransient` at the API service — idempotent at the OVH S3 layer, no caller-side state to pin. `putEncryptedShardFromFile` stays single-shot at the API service; the retry lives at the provider in a new private `retryShardCommit` helper that wraps stages 5-7 of `upload()` (startUpload → PUT → finishUpload) so the temp file, indexBytes, iv, fileKey, and hashHex all stay byte-identical across attempts. The wrap retries IOException and 5xx; never 4xx (logically-final). Capped at 3 attempts with simple exponential backoff (Retry-After honoured when present). startUpload is re-issued per attempt because OVH presigned URLs expire and a fresh descriptor is cheaper than wrapping the stale URL in an extra `getBridgeFileInfo` round-trip; shardUuid drift across attempts is safe because `commitWithRetry`'s 409 reconcile keys on plainName + size + start window, not the shardUuid. `downloadFileStreaming` also stays single-shot at the API service; the retry sits in the provider's `downloadWithFileUuid` and re-resolves the OVH download URL via `getBridgeFileInfo` plus re-constructs the `Cipher` per attempt (the existing one is single-use via `doFinal()`). Re-derived iv and fileKey are deterministic given the server-stored indexHex so re-derivation is safe. `TRUNCATE_EXISTING` on the destination open mode (unchanged) discards any partial prior attempt's bytes. **Also discharges the BACKLOG design constraint** "Internxt upload retry must pin indexBytes": option (a) chosen — retry stages 5-7 only, indexBytes computed once outside the retry boundary. One pinned IV-pinning test (`upload retry pins indexBytes across attempts (IV stability)`) added to `InternxtApiServiceTest`: drives a real `InternxtProvider.upload()` against a stateful MockEngine that fails attempt-1 PUT and attempt-2 finishUpload, then succeeds attempt-3. Asserts that across the two finishUpload calls the `indexHex` is byte-identical AND that the PUT ciphertext bodies are byte-identical across all three attempts (equal ciphertext under AES-CTR proves indexBytes/iv/fileKey all stable; any drift would change the keystream).

- Internxt request prioritization. Two-lane Foreground/Background overlay on top of `HttpRetryBudget` so user-driven traffic doesn't compete with scan metadata for the global concurrency budget. Lane signal travels via a new `Priority` coroutine-context element installed by the engine: `gatherRemoteChanges` and `gatherStreamingChanges` wrap in `Priority.Background`; Pass 1 (sequential mkdir/move/delete/rename) and the Pass 2 transfer launches wrap in `Priority.Foreground`. Strict priority: background callers spin (suspend, not busy-wait) on `delay(yieldWindowMs)` increments until no foreground slot has been claimed within the last `yieldWindowMs` (`= minSpacingMs`, 500 ms on Drive REST); when foreground is saturated, background starves. One global storm detector and one global spacing — both lanes feel a 503. `InFlightDedup` gains a priority-aware overload `load(key, priority, loader)`: the entry's effective priority is held in an `AtomicReference<Priority>` shared with the winner's loader via a new `Priority.Promotable` element, so a foreground caller joining an in-flight background load flips the reference to Foreground and the winner's spin loop breaks out the next tick (the resolve happens on each iteration). Promotion is monotonic — Foreground never demotes. Internxt call-site wiring: the four dedup'd reads (`getFolderContents`, `getFileMeta`, `listFiles`, `listFolders`) pass `currentPriority()` into the new overload; all other Drive / Bridge call sites already go through `awaitSlot()` which reads the context. Tests in `InternxtApiServiceTest`: foreground call jumps the queue ahead of parked background callers; dedup promotes a background load to foreground when a foreground caller joins (verified by `currentPriority()` read from inside the winner's loader at release); background starves under a continuously-claimed foreground over 60 s of virtual time and admits within `yieldWindowMs` when foreground stops; storm detector and spacing apply uniformly across lanes. - Internxt `finishUpload` idempotency via 409 reconcile. A network drop after the server-side commit but before the client reads the response previously caused a re-finish that orphan-billed a second `BucketEntry`; the Bridge exposes no client-addressable session-status endpoint to disambiguate. The fix layers a new internal `commitWithRetry(api, bucket, folderUuid, plainName, ext, fileSize, encryptedSize, indexHex, hashHex, shardUuid, startedAt, clock)` helper at the bottom of `InternxtProvider.kt` around the bridge `POST /v2/buckets/{bucket}/files/finish` call so the provider's upload pipeline now calls the helper at stages 7-8 instead of `api.finishUpload` directly. Zero matches but a stale name match exists → case (b), `delay(2s)` and re-list once before falling through to case (a). One match → `ReconcileResult.Found`, return the synthesized `BucketEntry(id = file.fileId, ...)`. Multiple matches → pick the youngest by `creationInstant` with a structured WARN naming both the chosen and rejected sibling fileIds (uses an SLF4J `ListAppender` in the new tests to assert the log shape). IV-pinning is honoured because the reconcile path holds `indexHex/hashHex/shardUuid` identity stable across the probe and the case-(a) re-attempt — same Bridge identity decrypts with the same key/IV as the original attempt. - OneDrive `fileSystemInfo` round-trip. Local mtime/ctime now survives an upload/download cycle on OneDrive instead of being replaced by the server's commit timestamp (which on `replace` uploads is always seconds-from-now and on permission changes drifts further). The fix has three legs. (1) Download/listing: `OneDriveProvider.toCloudItem` and the inline `listSharedWithMe` mapping prefer `fileSystemInfo.lastModifiedDateTime` / `.createdDateTime` over the top-level `lastModifiedDateTime` / `createdDateTime` when present (with `Instant.parse` wrapped in `runCatching` for both — defensive against a malformed timestamp string). Legacy items without the facet fall back to the top-level fields so existing rows keep working. (2) Upload (simple PUT, ≤ 4 MiB): `GraphApiService.uploadSimple` now accepts an optional `FileSystemInfo`; on the success path it follows the content PUT with a `PATCH /me/drive/items/{id}` carrying a `{"fileSystemInfo": {...}}` body. The content PUT cannot carry the facet inline so the two-step shape is forced by the API surface. Null fileSystemInfo skips the follow-up so call sites that don't care are byte-identical to before. (3) Upload (session, > 4 MiB): the `createUploadSession` body — formerly a fixed `{"item": {"@microsoft.graph.conflictBehavior": "replace"}}` literal — is built via `buildJsonObject`, and when fileSystemInfo is supplied the `item` block carries it alongside `conflictBehavior`. The session creation is a single round-trip so the file gets its timestamps stamped before the first chunk PUT lands. `OneDriveProvider.upload` reads `BasicFileAttributes` once at the top, formats `creationTime()` / `lastModifiedTime()` via `Instant.toString()` (produces ISO-8601 with `Z` suffix, exactly what Graph wants), and threads the `FileSystemInfo` into both branches. Filesystems without a real creation time (ext2, FAT) return `lastModifiedTime` for `creationTime` per JDK contract — we just send what we have; either both fields equal mtime, or both reflect reality. Four unit tests inside `ListChildrenPaginationTest.kt` (the existing GraphApiService MockEngine test) pin the upload wire shape: simple-PUT-with-fsi issues PUT then PATCH with both timestamps in the PATCH body, simple-PUT-without-fsi stays at one request, session-create body includes the `fileSystemInfo` block under `item` when supplied, session-create body omits it when null. One unit test inside `OneDriveProviderVaultFilterTest.kt` (the existing OneDriveProvider MockEngine test) pins the download precedence: a listChildren response containing one item with both top-level and fileSystemInfo timestamps and one item with only top-level surfaces `CloudItem.modified` / `.created` from fileSystemInfo for the first and from top-level for the second. Test-helper sidecar: a `drainOutgoingContent` that handles all four Ktor 3.x OutgoingContent subtypes (ByteArrayContent including DefaultTransformer-anonymous subclasses, TextContent, ReadChannelContent, WriteChannelContent) — Ktor's defaultTransformers wraps String/ByteArray bodies in anonymous `OutgoingContent.ByteArrayContent` subclasses that the top-level `io.ktor.http.content.ByteArrayContent` `is` check misses; the helper checks the nested abstract base.
- Internxt destructive-overwrite warning. Opt-in "rename-and-keep" mode compensating for Internxt's missing server-side versioning: when `[providers.<name>] keep_overwritten = true` is set, every replace renames the prior cloud file to `${plainName}.unidrive-prev-${utcStamp}` (UTC, `yyyy-MM-dd'T'HH-mm-ss`, `-` instead of `:` for NTFS portability) and creates the new content as a fresh row instead of dropping the prior bytes via `PUT /files/{uuid}`. Provider-scoped (Internxt only); `InternxtConfig` gains a `keepOverwritten: Boolean = false` field; `RawProvider` gains a nullable `keep_overwritten` TOML key; `InternxtProviderFactory.create` threads the property through; `Main.rawProviderToProperties` exposes it on the properties map. The upload pipeline's existing `if (existingRemoteId != null)` branch now consults the config: on opt-in it calls a new private `tryKeepOverwrittenRename` helper that resolves `(plainName, type)` via `getFileMeta(existingRemoteId)`, computes the archive name, and walks the counter suffix (`-2`, `-3`, ...) on 409 collisions up to `MAX_KEEP_OVERWRITTEN_RETRIES = 5`; success replaces the `replaceFile` call with `createFile(plainName=original, ...)`; failure (collision-exhausted, 5xx on rename, getFileMeta IO error) logs WARN and falls through to the destructive `replaceFile` rather than stranding the local edit. Encrypted `name` field on the archived row stays bound to the original plainName (only `PUT /files/{uuid}/meta` is called, which updates `plainName` and `type` but not the encrypted `name`) — documented in the README; not fixed because the encrypted name is only used by the server-side disambiguation and `getMetadata`/`listChildren` already resolve via `plainName`. Orthogonal to `ConflictPolicy` by construction (KEEP_BOTH emits a new-path Upload with `remoteId = null`, never reaching the `existingRemoteId != null` branch; LAST_WRITER_WINS where remote loses emits an Upload with `remoteId = existing` which is the only path that consults `keepOverwritten`). Storage cost doubles per edit with no automatic prune — the `unidrive prune --backups` CLI is filed as a separate BACKLOG entry under Low.
